# AI Summary Spec

Each analysis ends with a natural-language coach report generated by OpenAI
(`gpt-5-mini`). This document specifies the prompt structure, the inputs, the
expected output, and the guardrails.

## Why this exists

Raw metrics + 0–100 scores are useful but cold. Coaches want a paragraph that
says *what the numbers mean* for this athlete. The AI summary is that
paragraph — bounded, factual, and grounded in the metrics it received.

## Model selection

- **Default (all reports)**: OpenAI `gpt-5-mini`
- **Fallback (only if quality is insufficient — Phase 6 follow-up)**: `gpt-5`

The choice is configured per environment, not per call. Production goes
through the OpenAI client in `src/ai_summary/summarizer.py`. The API key is
loaded from `.env` (`OPENAI_API_KEY`) via `python-dotenv` at entrypoints only
— library code never reads `.env` directly.

## Expected output structure

A single JSON object with three sections — kept short on purpose so it fits
on a results page without scrolling.

```json
{
  "headline": "Strong overall sprint, with acceleration the standout phase.",
  "strengths": [
    "Maximum velocity (7.2 m/s) sits in the top quartile for U18 male athletes.",
    "Time to 10 m is sharper than time to 20 m, suggesting effective first-step quickness."
  ],
  "areas_to_develop": [
    "The gap between 20 m and 30 m splits suggests velocity is held but not extended — consider top-speed maintenance work."
  ],
  "training_suggestions": [
    "Flying 20s and 30s with 3-min rests to extend top-speed phase.",
    "Resisted sprints (≤10% BM) to reinforce acceleration mechanics."
  ]
}
```

Strict bounds:
- `headline`: ≤ 120 characters
- `strengths`: 1–3 bullets, each ≤ 200 characters
- `areas_to_develop`: 1–3 bullets, each ≤ 200 characters
- `training_suggestions`: 1–4 bullets, each ≤ 200 characters

The summariser validates output against this schema and retries (up to 2 ×)
if the model overshoots.

## Prompt template structure

Templates live in `src/ai_summary/templates/<family>.md`. One per CV family —
**eight** total: sprint, agility, jump, dribbling, endurance, throw, skill,
mobility (LESS only) — **not** one per test. Test-specific nuance comes
through the input JSON, not by forking the template. Cognitive tests,
Sit-and-Reach, and Stepwise Core Stability are out of v1 CV scope and have
no template here.

A template has four blocks:

1. **System role** — "You are a sports-science analyst writing a brief..."
2. **Domain framing** — vocabulary for that family (e.g. "first-step quickness", "eccentric load tolerance")
3. **Cross-metric narratives** — pre-defined patterns the model should recognise (see below)
4. **Hard guardrails** (always, every template):
   - No medical advice or diagnoses
   - No injury-risk pronouncements
   - No comparisons to specific named athletes
   - Cite numbers from the input — do not invent values
   - Reference the benchmark band, not absolute claims about quality
   - If `benchmark_confidence == 'provisional'`, soften language ("preliminary indication")
   - Output exactly the JSON schema; no preamble, no markdown fences

## Cross-metric narratives (per family)

These are patterns the prompt instructs the model to look for:

**Sprint**
- High max speed but slow 10 m → top-end strong, acceleration weak
- Strong 10 m but slow 30 m → acceleration good, speed maintenance weak
- High peak deceleration but low max speed → braking-limited

**Jump**
- Low CMJ but high RSI in drop jump → poor concentric force, good elastic ability
- Low ground contact time + low jump height → drop height too high for athlete

**Dribble**
- Low touches/m + high ball–foot distance → loose control
- High touches/m + low max speed → control prioritised over speed

**Agility**
- Strong in straight segments, weak in COD → deceleration / re-accel limited
- Symmetric splits but slow overall → conditioning, not technique

The full list is in each family's prompt template.

## Caching & determinism

- Set `temperature=0.3` for consistency
- Use OpenAI structured outputs (JSON schema) so the response shape is enforced model-side; the local validator is a second line of defence
- Cache by hash of input JSON (results layer); identical inputs return identical summaries
- Failed calls do not block the rest of the report — return `summary: null` with an `error` field; the UI renders the metrics + scores anyway

## Implementation

- `src/ai_summary/summarizer.py` — the OpenAI client (retries, validation, caching, cost log line)
- `src/ai_summary/prompts.py` — assembles system + user prompts from template + input
- `src/ai_summary/templates/*.md` — the seven per-family templates
- `tests/unit/ai_summary/` — mock the OpenAI client; test prompt assembly + output validation. CI must never hit the network.
